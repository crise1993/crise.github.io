    "jsx": "preserve",                     // 指定 jsx 代码的生成: 'preserve', 'react-native', or 'react'

    "sourceMap": true,                     // 生成相应的 '.map' 文件
    "outFile": "./",                       // 将输出文件合并为一个文件

# compilerOptions详解

# baseUrl

类型：字符串；默认值："./"。

引入非绝对地址模块的基地址。当moduleResolution属性值为node的时候，如果我们引入了一个非相对模块，那么编译器只会到node_modules目录下去查找，但是如果配置了baseUrl，那么编译器在node_modules中没有找到的情况下，还会到baseUrl中指定的目录下查找；

示例：

| package.json | tsconfig.json |
---src | app.ts | ---views ---list Bar.ts ---utils index.ts

在上面的目录结构中，我们如果在Bar.ts文件中引用utils/index.ts文件中的内容，则需要写很长的路径：

import { toString } from '../../utils/index.ts';

我们能否以 src 为默认根路径来引用内容呢？

import { toString } from 'utils/index.ts';

我们直接尝试这么做会TS会提示can't find ...的报错，此时我们在tsconfig.json中加入baseUrl: "./src"，

{
  "compilerOptions": {
    "baseUrl": "./src"
  }
}

会发现TS不再提示报错。（tsconfig.json 中的配置只是起提示作用，但是当真正在查找文件时，得在相应的打包工具里配置别名，没有配置 alias，还是会报错）

# paths

类型：对象。

这个是配合baseUrl一起使用的，因为其是相对于baseUrl所在的路径的，主要用于到baseUrl所在目录下查找的时候进行的路径映射。

示例：

{
  "compilerOptions": {
    "baseUrl": "."
    "paths": {
      "@/*":["src/*"],
      "jquery": ["node_modules/jquery/dist/jquery"] 
    }
  }
}

如果配置了paths(至少配置了一项)，那么TS编译器其只会到baseUrl/映射路径下查找。

在这个例子中，

  • 第一条配置使得无论当前编写的文件在目录结构中的什么位置，或是与src中目标文件的相对关系如何，都可以直接使用@前缀且相对于baseUrl的路径来访问；
  • 第二条配置，显示的设置了jquery的位置。

# typeRoots

类型：字符串数组；默认值："["node_modules/@types"]"。

这个用于指定类型声明文件的查找路径。默认在node_modules下的@types里面查找。需要注意的是这里仅仅是d.ts文件的查找路径。同样，这个也是相当于在引入非相对模块的时候拓宽了类型声明文件的查找范围，其实就是配置类型声明文件的查找目录。

示例：

{
  "compilerOptions": {
    ...
    "typeRoots": [
      "node_modules/@types",
      "./typings"
    ]
  }
}

# allowSyntheticDefaultImports

类型：布尔。默认：false。

将全部导出转换为默认导出。

如果一个模块，没有默认导出，那我们需要已这样的方式引入全部变量：

import * as _ from 'lodash'
// 设置了"allowSyntheticDefaultImports": true后，我们可以以默认导出的方式引入模块：
import _ from 'lodash'

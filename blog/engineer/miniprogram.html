# 小程序跨端方案

# 1、背景

目前，小程序在用户规模及商业化方面都取得了极大的成功。除了微信之外，支付宝、京东、百度、字节跳动等平台都推出了自己的小程序。

每一家小程序都重新开发一遍显然不现实，各大平台的流量和优于H5的体验也不可能放弃，因此探索一个小程序的跨端方案，以最小的成本，实现在微信、快手、百度等多个小程序上的业务快速迭代，势在必行。

# 2、 调研

# 2.1 公司内目前多端处理方案

跨端框架 uni-app、taro等  原生直接写 原生壳+webview

# 2.2 市面上小程序跨端方案

小程序跨端在各大公司都有强烈的需求，发展至今已百花齐放，下面罗列了一些各大厂商提供的框架：

uni-app taro remax mpvue chameloen megalo kbone 厂家 Hbuilder 京东 蚂蚁金服 美团 滴滴 网易 微信 从轻轻松松就罗列这么一堆框架，其实可以看出一些东西：

市场需求确实很大， 目前没有一家框架是有压倒性优势的，而市场需求就那么大，肯定是支撑不了这么多框架持续繁荣下去的，后面肯定会有不止一个框架会慢慢死掉 因为这是一个兼容性质的框架，是需要跟随各大厂商API变动，即时更新的

公司不是开源组织，不可能存在脱离公司需求的市场需求。开源是他们的一个态度和对社区做的贡献，谁也不敢保证哪个框架可以一直高效的迭代下去

一旦框架停止维护，那使用这个框架的项目再想改换别的方案，就很难了

（ps：mpvue目前已停止更新）

# 2.3 瓜子的解决方案

基于市场的调研，以及瓜子目前业务快速迭代的现状，公司内已初步达成统一意见：小程序多端适配使用H5+webview的形式

这个方式可大大缩减开发成本，但一个h5项目，想要同时在多个小程序内正常运行，还是有不小的兼容成本，这个成本主要在两方面：

每个小程序需要一个原生壳，以及一些功能的承接逻辑，比如登录、支付、webview、地图等等 h5里使用原生小程序的功能，需要区分小程序判断使用，以及一些展示上的判断 而进一步缩减这些成本，并且统一规范，就是跨端SDK要做的事情

# 4 使用方式

使用分为新项目和老项目两种场景

# 4.1 新项目接入步骤

1）拷贝3.2.1里标准代码

2）更改小程序相关配置（appid、域名配置、底部菜单配置...）

3）安装依赖，执行编译

4）h5项目做必要的小程序适配，比如登录等功能

测试无问题后即可发布到对应小程序平台

# 4.1 老项目接入步骤

1）拷贝3.2.1里项目的script文件夹到业务项目根目录

2）调整 script/config 内的目录配置匹配业务项目

3）安装相关依赖，执行编译脚本

4）将原项目中微信小程序内置API替换为webview-sdk提供的API

测试无问题后即可发布到对应小程序平台

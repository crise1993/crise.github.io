# 浏览器渲染原理

本质上，浏览器的渲染过程，就是根据描述或定义构建数学模型，通过模型生成图像的过程。而当用户与页面发生交互动作时，则显示新的图像。

上图是经典的webkit渲染流程，不同的浏览器渲染引擎，渲染过程不完全相同，但大致流程都差不多。然而伴随着现代浏览器的发展，渲染引擎架构也相应的发生了一些变化。

以现在的Chrome浏览器为例，浏览器渲染页面的过程大致分为以下6个步骤：

    1. 根据HTML解析生成DOM树
    1. 根据CSS解析生成CSS规则树
    1. 结合DOM树和CSS规则树，生成渲染树
    1. 布局渲染树，生成Layout Tree
    1. 分层布局树，生成Layer Tree
    1. 绘制与合成

前面5个步骤都是在渲染进程中的 主线程 中进行的，最后一步则由渲染进程的合成线程、栅格线程完成图像的绘制工作，最终GPU合成页面后，drawing 到屏幕上。

# 1. 根据HTML解析生成DOM树

浏览器解析html是从上往下一行一行地解析的。当解析到script标签时，会暂停DOM构建，等待脚本下载并执行完毕，才会继续向下解析。js脚本会阻塞DOM解析，进而影响页面渲染速度。

当整个解析的过程完成以后，浏览器会通过DOMContentLoaded事件来通知DOM解析完成。

# 1.1 子资源加载

在构建DOM的过程中，会解析到图片、CSS、JavaScript脚本等资源，这些资源是需要从网络或者缓存中获取的，主线程在构建DOM过程中如果遇到了这些资源，逐一发起请求去获取，但是为了提升速度，浏览器会运行预加载扫描器（preload scanner）对HTML文档进行快速扫描（这个扫描过程不需要完全解析整个HTML文档，而是仅识别文档的基本结构和关键元素），如果HTML中存在img、link、script等标签，它会查看html解析器生成的Token，解析到相关文件之后，预加载扫描器会把这些请求传递给Network Process提前进行资源下载。

# 1.2 解析DOM流程

解析的过程可以分为以下四个步骤：

# 解码（decoding）

服务器主机传输回来的其实都是一些二进制字节码，浏览器需要根据文件charset指定编码（例如UTF-8）转换成字符串，也就是HTML代码。

# 符号化（Tokenization）

浏览器容错进制：你从来没有在浏览器看过类似"语法无效"的错误，这是因为浏览器去纠正错误的语法，然后继续工作。

符号化是词法分析的过程，将字符串解析成符号（Token），HTML 符号包括：开始标签(StartTag)、结束标签(EndTag)、文本标签(textTag)。

Token分为 Tag Token 和 文本 Token。HTML 解析器开始工作时，会默认创建一个根为 document 的空 DOM 结构，同时会将一个 StartTag document 的 Token 压入栈底，然后进行不断压栈出栈操作（碰到StartTag会入栈，碰到EndTag会出栈），直到栈空完成解析。

具体的词法分析过程可以查看符号化及构建树的完整算法 (opens new window)。

# 生成Nodes并构建DOM树（tree construction）

ps：符号化和生成Node是并行操作的，也就是说一边进行词法解析生成Token，一边采取深度遍历算法来创建一个 Node 节点。

浏览器会根据Token里记录的 开始标签和文本标签，创建Node节点（Node节点包含了html标签的所有属性），然后渲染引擎会把Node节点以合适的方法插入到DOM树中，当Token栈清空完毕，紧接着DOM树也就构建完成。

# 2. 根据CSS解析生成CSSOM

# 2.1 概念

# CSS 继承

假如body有一个样式规则是font-size: 16px，之后，body上每个子节点都会继承font-size: 16px这条样式规则。每个子节点都可以在这个样式基础上还会添加自己单独的样式规则，比如span又添加了一个样式规则font-size: 14px。正是因为样式这种类似于继承的特性，浏览器设定了一条规则：CSSOM Tree需要等到完全构建后才可以被使用，因为后面的属性可能会覆盖掉前面的设置。

# CSS 层叠

在匹配一个节点对应的 CSS 规则时，是按照从右到左的顺序的，例如：div p { font-size :14px } 会先寻找所有的 p 标签然后判断它的父元素是否为 div。

所以我们写 CSS 时，尽量用 id 和 class，千万不要过度层叠。

# CSS 级联

浏览器需要一种方法来确定哪些样式才真正需要应用到对应元素，所以它使用一个叫做css specificity (opens new window)的公式，这个公式会通过：

  • !important，特性值最大，优先性最高
  • 特性值大的优先
  • 特性值如果一样，那么就近原则

对于第二点特性值的具体计算如下：

属性 特性值
行内样式表 1000
id选择器 100
类选择器、属性选择器、伪类选择器 10
标签选择器 1
层级选择器（空格、>、+、~、*） 0

然后相加得出一个权重值，取最高的那个。

# 2.2 解析流程

构建CSSOM的过程与构建DOM的过程非常相似，当css解析器接收到网络进程返回的text/css类型字节流之后，然后解码成CSS文本字符串，再进行解析、标准化转换、符号化（Tokenization），然后构建Nodes节点最后生成CSSOM。

在这一过程中，浏览器会确定下每一个节点的样式到底是什么，并且这一过程其实是很消耗资源的。因为样式你可以自行设置给某个节点，也可以通过继承获得。

# 格式转换

解码之后，CSS解析器 会根据语法规范 (opens new window)，对CSS文本字符串（link引用的css文件、style标签内的CSS、元素style属性内嵌的CSS）执行一个解析操作，将CSS文本字符串转换成浏览器可以理解的结构 — 样式表-Style Sheets (opens new window)。

# 标准化（Normalization）属性值

转换样式表中的属性值（如颜色、尺寸等），使其标准化（Normalization）。

# 符号化（Tokenization）

符号化是词法分析的过程，将 样式表 解析成符号（Token） 。在词法分析阶段，CSS 代码被分解为一系列的令牌（Tokens），每个令牌代表 CSS 代码的一个部分，如选择器、属性名、属性值、分号、大括号等。

# 生成Nodes并构建CSSOM（tree construction）

语法分析（Syntax Analysis）：在语法分析阶段，词法分析生成的令牌被组织成语法树（Parse Tree）或抽象语法树（AST）。这个树状结构表示了 CSS 代码的层次结构和语法关系。

这个阶段会经历 语法分析，根据css specificity (opens new window)计算选择器特异性，根据CSS继承和层叠规则等计算标签节点样式，最后生成CSSOM。

# 2.3 为什么需要CSSOM

CSSOM (opens new window) 是由 CSS 文本解析得到的渲染引擎能够识别的结构，类似 HTML 和 DOM 的关系，CSSOM 树描述的是选择器之间的层级关系，可以为 JavaScript 提供操作样式表的能力，还能为布局树的合成提供基础样式信息。体现在 DOM 中即 document.styleSheets。

CSSOM是依赖DOM树的，既然已经有了DOM树结构来表示HTML文档，那为什么不把CSS顺便放在DOM上，以便我们直接从Element上获取所有样式信息呢？

因为如果把CSS信息一起建模在DOM结构上，就会违背单一职责原则。因为正如在网页中HTML承担了语义职能，CSS承担了表现职能一样，在计算机中DOM承担了语义职能，而CSSOM承担了表现职能。

# 3. 生成渲染树

DOM树和CSS规则树全部准备好了以后，浏览器才会开始构建渲染树。其实这就是一个 DOM 树和 CSSOM 树合并的过程。

在这一过程中，浏览器得递归 CSSOM 树，然后确定具体的元素到底是什么样式（CSS匹配HTML元素是一个相当复杂和有性能问题的事情。所以，DOM树要尽量小，CSS尽量使用id和class，不要过度层叠）。

# 3.1 工作流程

渲染树的重要特性是它仅捕获可见内容，构建渲染树浏览器需要做以下工作：

  • 从 DOM 树的根节点开始遍历每个可见节点（应用了具有类似内容的伪类 p::before{content:"Hi!"} 也会包含在渲染树中，即使它不在DOM中）；
  • 有些节点不可见（<html>、<script>、<link>等），因为它们不会体现在渲染输出中，所以会被忽略；
  • 某些节点被CSS隐藏（设置了display: none属性，visibility: hidden则不会），因此在渲染树中也会被忽略；
  • 对于每个可见节点，为其找到适配的 CSSOM 规则并应用它们。

# 3.2 延伸阅读

# CSSOM 会阻塞生成渲染树

CSS 加载不会阻塞 DOM 树的解析，但会阻塞 渲染树 的生成（解析白屏），即阻塞页面的显示，因为需要等待构建 CSSOM 完成后再进行构建布局树。

# CSS 会阻塞JavaScript执行

我们都知道，在解析html构建DOM的过程中，JS的加载、解析与执行，会阻塞DOM的构建过程。因为 JavaScript 可能会修改当前状态下的 DOM，如果不阻塞，那么这边在构建DOM，那边JavaScript在改DOM，如何保障最终得到的DOM是否正确？因此，为了防止渲染出现不可预期的结果，所以需要阻塞 DOM 解析。

但是，JavaScript对渲染过程的影响不只是阻塞DOM的构建，它会导致CSSOM也阻塞DOM的构建。因为 JavaScript 具有修改 CSSOM 的能力，所以在js脚本执行前，如果发现页面中包含 CSS样式 (外部文件引用或内置 style 标签) 还需要等待CSSOM的生成。

# 4. 布局渲染树，生成Layout Tree

重排/回流（reflow）：某个元素的尺寸、位置、内容、字体大小等发生了变化，则需重新计算渲染树，重新渲染。reflow基本是无法避免的，因为当你滑动一下鼠标、resize 窗口，页面就会产生变化。

“布局”就是计算渲染树中所有节点的几何属性，比如：位置、大小、边框、形状等。

# 4.1 工作流程

确定页面的布局是一项耗时的任务。即使是最简单的页面布局，例如：从上到下的流式布局，也必须考虑字体有多大以及在哪里换行，因为这些会影响段落的大小和形状，这会影响下一段的位置；CSS 可以使元素向一侧浮动、屏蔽溢出项以及改变书写方向。

可以想象，这个布局阶段的任务是艰巨的。在 Chrome 中，有一整个工程师团队负责布局模块。如果你想具体了解其中的工作细节，可以点击查看few talks from BlinkOn Conference (opens new window)。

# 4.2 延伸阅读

Dirty bit系统 (opens new window)：为了不因为每个小变化都全部重新布局，浏览器使用一个dirty bit系统，一个渲染对象发生了变化或是被添加了，就标记它及它的children为dirty——需要layout。存在两个标识——dirty及children are dirty，children are dirty说明即使这个渲染对象可能没问题，但它至少有一个child需要layout。

# 5. 分层布局树，生成Layer Tree

渲染层（RenderLayers） 的存在是为了使页面元素以正确的顺序组合，以正确显示重叠内容、半透明元素等。可以在DevTools的Performance —> Layers面板，查看网站具体是如何划分层的。

布局渲染树后，渲染引擎并不是直接使用渲染树进行绘制，渲染引擎还要遍历布局树，为特定的节点生成专用的图层（RenderLayer），并生成对应的图层树（Layer Tree）。

# 5.1 生成规则

布局树的节点默认从属于父节点图层，满足以下两点中任何一点的元素可被单独提升为一个图层：

# 拥有层叠上下文属性

明确定位（position）、透明（opacity）、CSS 滤镜（filter）、transform 动画等拥有层叠上下文 (opens new window)属性的元素。

# 需要剪裁(clip)的内容

当内容展示不下被隐藏或出现滚动条时，内容区域 会单独创建一个图层。

# 5.2 延伸阅读

合成层优化：在Chrome的渲染机制中，图层分成两种，除了渲染层，还有合成层（Compositing Layers）。渲染层可以提升成合成层，提升成合成层后，便不会影响到其他元素的绘制。

具体 合成层提升规则、工作原理，优化以及调试手段等，可以点击查看。

# 6. 绘制与合成

绘制（paint）和分层是同时发生的，也就是说主线程遍历布局树生成Layer Tree的同时，也会生成绘制记录。所以有的文章或书籍将分层放在绘制之后，虽然不完全准确（因为绘制是一个多步骤的过程，下面会详细说明），但也可以理解。

# 6.1 概念

# 图块（Tile）

合成线程会将每个图层划分为若干个图块，图块大小通常为256x256或512x512。

# 位图（BitMap）

位图 是一种图像存储方式，将图像划分为一个像素（图像的最小单元）矩阵，每个像素通常由颜色信息（如红、绿、蓝）和可能的其他信息（如透明度或深度值）组成。

想像一下如果要绘制出一张图片，你应该怎么做，显然首先是把这个图片表示为一种计算机能理解的数据结构：用一个二维数组，数组的每个元素记录这个图片中的每一个像素的具体颜色。所以浏览器可以用位图来记录他想在某个区域绘制的内容，绘制的过程也就是往数组中具体的下标里填写像素而已。

所有的位图（包括图片、文本、栅格化的内容），最终都要由内存发送到显存，绑定为 GPU Texture（GPU纹理/贴图）。不论是提交到显存的过程，还是 GPU调整和渲染Texture的过程，都要消耗不少 GPU 资源。

# 显存（GPU Video RAM）

通常称为视频内存或显存，是一块专门的内存，用于存储图形和视频数据，以供图形处理单元（GPU）进行快速访问和处理。

显存用于存储屏幕上的 像素数据、纹理、渲染缓冲区、帧缓冲区和其他与图形相关的数据。这些数据需要快速访问，以便在屏幕上渲染图像。

# 纹理（Texture）

纹理其实就是存储在GPU中的位图。GPU纹理存储，需要有固定格式，如R5G6B5、A4R4G4B4等像素格式，便于兼容与处理。

另外 GPU 对纹理大小有限制，比如长/宽必须是2的幂次方，最大不能超过2048或者4096等。

# 绘制记录（Paint Records）

通常也被称为绘制调用记录（the recording of draw calls），或者绘制指令列表，里面包含了，用来记录绘制顺序（例如先绘制背景，然后是文字，然后是形状..）。

绘制记录序列化后会被存入到SkPicture中。

# SkPicture

合成线程上的 SkPicture 是一个可序列化的数据结构，可以捕获然后重播命令。绘制（paint）第1步中所生成的绘制记录会序列化进入其中，然后在光栅化时 replay 绘制调用命令。

SkPicture记录可以通过以下两种方式之一转换为 GPU 上的位图：

  • 由 Skia 的软件光栅化绘制为位图并作为纹理上传到 GPU；
  • 由 Skia 的 OpenGL 后端 (Ganesh) 直接绘制为 GPU 上的纹理。

# 光栅化（Rasterize）

回放绘制记录，执行绘制命令，将图层的图块转化成位图的过程叫做光栅化。视口附近的图块会优先生成位图，合成线程可以决定栅格线程光栅化图块的优先级，这样可以保证用户能看到的部分可以先被光栅化。一个层也会包含多种块以支持类似缩放这样的功能。

在纹理里填充像素不是那么简单的自己去遍历位图里的每个元素然后填写这个像素的颜色的。光栅化的本质是坐标变换、几何离散化，然后再填充。

光栅化包括软件光栅化和硬件光栅化：

  • 软件光栅化（Software Rasterization）

    基于CPU、使用Skia库，首先在CPU中绘制像素进位图，然后再作为纹理上传至GPU显存。

    软件光栅化可能非常昂贵，因此这种光栅化不会发生在合成线程中（它可能会阻止为活动树绘制新帧），而是发生在栅格线程中。可以使用多个栅格线程来加速软件光栅化。

  • 硬件光栅化（Hardware Rasterization）

    也就是我们常说的GPU Raster。基于GPU、借助Skia’s OpenGL backend (Ganesh)直接在GPU中进行光栅化，绘制填充像素，生成纹理。

    Ganesh光栅化图层时，SkPicture使用 Ganesh 回放绘制记录，生成 GL命令流，然后通过命令缓冲区传递给 GPU进程。当合成线程决定光栅化任何图块时，生成 GL命令 并立即发送给 GPU进程，并且将 图块 捆绑在一起以避免 GPU 上的图块光栅化产生过高的开销。

使用 硬件光栅化 的好处在于：以往 软件光栅化 的方式，受限于 CPU 和 GPU 之前的上传带宽，把 位图 从 RAM 里上传到 GPU 的 VRAM（显存） 里的过程是有不可忽视的性能开销的。若 光栅化 的区域较大，那么使用 软件光栅化 很可能在这里出现卡顿。

现在基本最新版的几大浏览器都是硬件光栅化了，但是对于一些移动端基本还是软件光栅化较多。打开chrome浏览器输入chrome://gpu/ 可以查看GPU加速情况。

# 绘制（paint）

绘制通常表示在图层表面上进行的一系列操作。

有时您可能会听到与绘制一起使用的术语“光栅化”。这是因为绘制实际上是两个任务：

  • 1）生成绘制记录（这个过程和遍历布局树生成layer Tree是同时发生的）；
  • 2）光栅化（回放绘制记录，执行绘制命令，填充像素：包括绘制文字、颜色、图像、边框和阴影等，也就是一个 DOM 元素所有的可视效果）。

# Draw Quads

是一组包含位图在GPU显存中的具体存储位置、以及将这些位图drawing在页面哪个位置的数据信息，用于描述如何使用位图合成页面。

# 合成帧（Compositor Frame）

存储了页面某一帧中关于Draw quads的集合。合成帧用于将所有纹理合并成一个大的位图。

# 合成（composite）

合成是一种将页面的各个部分分成图层、分别光栅化并在合成线程中创建合成帧，最终合成为页面的技术。如果发生滚动，由于图层已经光栅化，它所要做的就是合成一个新帧。动画则可以通过移动层来实现。

# 帧缓冲区（Frame Buffer）

帧缓冲区是计算机图形学中的一个重要概念，通常指的是用于存储图像或帧的内存区域。它在图形渲染和显示过程中扮演关键角色，允许计算机生成和显示图像。

在渲染过程中，计算机生成图像的每一帧（包括三维渲染、图像处理、动画等应用），通常首先被放入到帧缓冲区中。一旦图像被绘制到帧缓冲区中，它可以被传送到显示设备（通常是计算机显示器）上显示给用户。这个过程通常以连续的方式进行，以创建平滑的动画或视频。

# draw

表示GPU最终把纹理显示到屏幕上的过程。

# 6.2 工作流程

具体工作流程如下：

    1. 主线程 遍历生成layer Tree的同时，会生成绘制记录，然后主线程会将各个图层和绘制记录commit到合成线程（其中绘制记录会序列化存入SkPicture）；
    1. 合成线程将每个图层分成若干个图块（tile），然后发送给 栅格化线程池， 多个栅格线程并行执行SkPicture记录中的绘制命令，分别光栅化图块，并将转换后的位图上传到GPU并作为纹理存储；
    1. 所有图块光栅化完成后，合成线程收集Draw Quads信息并创建合成帧，然后合成帧通过IPC管道被发送给浏览器进程；
    1. 浏览器进程接收到合成帧之后，便将 合成帧 发送给 GPU，GPU根据 合成帧 进行合并渲染（对显存中所有纹理合并成一个最终的位图）。这时如果浏览器UI有变化，或者插件的UI有变化时，另一个合成帧就会被创建出来。所以每当有交互发生时，合成线程就会创建更多的合成帧然后通过GPU将新的部分渲染出来；
    1. GPU 渲染完成后将渲染结果放入帧缓冲区，视频控制器会按照 VSync（垂直同步） 信号逐行读取帧缓冲区的数据（接收到Vsync信号，这一帧开始），经过可能的数模转换传递给显示器显示。

以上步骤2、3、4是软件光栅化的完整流程。而硬件光栅化时，SkPicture 使用 Ganesh 回放绘制记录，直接在GPU中进行光栅化生成纹理，然后GPU Thread调用平台对应的3D API(windows下是D3D，其他平台都是GL)，进行纹理的合并。GPU合并渲染完成后，再进行步骤5。

在纹理的合并时，借助于3D API的相关合成参数，可以在合并前对纹理Transformations（也就是之前提到的位移、旋转、缩放、alpha通道改变等等操作），先变形再合并。

# 6.3 延伸阅读

重排、重绘与合成：而渲染流水线任意一帧的生成方式，有 重排、重绘和合成三种方式，按照效率推荐合成方式优先，在不能满足需求时使用重绘甚至重排的方式。这篇文章重点放在讲解浏览器重排、重绘与合成的概念、渲染区别、优化手段等。

# 7. 接收到用户输入浏览器怎么处理？

从浏览器的角度来看，输入意味着用户的任何手势。文本框中键入/单击鼠标是输入，鼠标滚轮滚动是输入，触摸/鼠标悬停也是输入事件。

# 7.1 工作流程

# 发送事件类型及坐标

当用户手势（例如触摸屏幕）发生时，浏览器进程是第一个接收该手势的进程。然而，浏览器主进程只知道手势发生的位置，因为选项卡内的内容是由渲染进程处理的。因此，浏览器进程需要将事件类型（如click）及其坐标发送到渲染进程。渲染进程通过查找事件目标（event target）并运行附加的事件侦听器（event listener）来适当地处理事件。

# 检查非快速滚动区域

由于运行 JavaScript 是主线程的工作，因此当合成页面时，合成线程会将附加了事件处理程序的页面区域标记为非快速滚动区域（non-fast scrollable region）。 合成线程检查输入事件的坐标是否为 非快速滚动区域。如果事件发生在该区域，合成线程可以确保将输入事件发送到主线程。如果输入事件来自该区域之外，则合成线程将继续合成新帧，而无需等待主线程。

# 查找事件的目标对象

当合成线程将输入事件发送到主线程时，首先要运行的是命中测试（hit test）来查找事件目标。命中测试 使用渲染过程中生成的绘制记录数据来找出输入事件发生的x, y坐标点上对应的是哪个对象元素。然后触发目标对象绑定事件的JavaScript执行。

# 7.2 慎用body的事件委托

Web 开发中有一种常见的事件处理模式是事件委托（event delegation）。由于事件的冒泡，您可以给顶层元素（如body）绑定一个事件监听函数来作为其所有子元素的事件委托者，然后根据事件目标进行任务处理。您可能已经看到或编写过如下代码。

document.body.addEventListener('touchstart', event => {
  if (event.target === area) {
    // todo
  }
})

这种为所有元素绑定一个事件监听函数的写法看上去代码很简洁。可是，如果从浏览器的角度去审视该代码，你会发现整个页面都标记成了一个非快速滚动区域。这就意味着即使你页面的某些区域压根不在乎是不是有用户输入，当用户输入事件发生时，合成线程每次都会与主线程进行通信，并且会等待主线程处理完js任务才干活。因此这种情况下，合成线程就丧失了平滑滚动能力（smooth scrolling ability）。 为了减轻这种情况的发生，您可以在事件侦听器中添加passive: true选项。这个选项会告诉浏览器你仍希望在主线程中监听事件，但合成线程也可以继续合成新帧。

document.body.addEventListener('touchstart', event => {
  if (event.target === area) {
    // todo
  }
}, {passive: true})

# 7.3 最小化向主线程调度事件

显示器的刷新频率通常是 60/s，我们可以通过让JS代码的执行频率和屏幕刷新频率保持一致来实现页面的平滑动画（smooth animation）效果。对于用户输入来说，触摸屏一般每秒传送触摸事件 60-120 次，而鼠标则会每秒传送事件100次，因此输入事件的触发频率其实远远高于我们屏幕的刷新频率。

如果像touchmove这样的连续事件每秒发送到主线程 120 次，那么与屏幕刷新速度相比，它可能会触发过量的命中测试和 JavaScript 执行。 为了最大限度地减少对主线程的过多调用，Chrome 会合并连续事件（如wheel、mousewheel、mousemove、pointermove、touchmove等），并将调度延迟到下一个requestAnimationFrame之前。 任何离散事件（如keydown、keyup、mouseup、mousedown、touchstart、touchend、click等相对不怎么频繁触发的事件）都会立即调度。

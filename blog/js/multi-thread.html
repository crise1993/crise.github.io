# JS中的多线程编程

有些人看到标题可能会立马产生疑问：JS 不是单线程嘛？JS中关于IO/异步阻塞（setInterval/setTimeout、Promise、requestAnimationFrame, queueMicrotask等）不是通过事件循环（Event Loop）来解决的吗？

的确，JS作为一门动态、解释型语言，是单线程的，所有的代码都由“主线程”依次执行，IO/异步阻塞问题通过事件循环来解决。这种单线程模型带来了一些优点，比如简单、易于理解、避免了死锁等问题。然而，随着Web应用程序的复杂性和性能需求的增加，这套机制当线程需要处理大规模的计算的时候就不大适用了。

# 单线程模型的局限性

单线程模型有明显的局限性，主要包括以下几个方面：

  • 阻塞主线程：耗时任务会阻塞主线程，导致页面卡顿或失去响应，用户体验下降；
  • 限制CPU利用率：JS单线程模型无法充分利用多核CPU的优势，无法实现真正的并行计算；
  • 无法进行密集型计算：大量的计算任务会占用主线程，影响页面的渲染和交互；

试想一下某个场景：

  • 你需要实现对文件的加解密；
  • 你的 VirtualDom树有很多元素（例如上万个），你需要对这棵树进行 Diff操作；
  • 你需要在浏览器“挖矿”。

上面这些场景都会阻塞主线程，也就是当进行这些操作的时候，你的页面大概率会卡住，而且当页面卡住一段时间后，Chrome 等浏览器会 Kill 掉当前的Tab进程。为了解决这些问题，JS引入了多线程编程模型，使得开发者可以利用多个线程并行执行任务。本文将深入探讨JS中的多线程编程技术，包括Web Worker、SharedArrayBuffer、Atomics等，并介绍如何在实际项目中应用这些技术来提高性能和响应能力。

# Web Worker

Web Workers是一种在后台运行JS代码的机制，它允许在单独的线程中执行脚本，而不会阻塞主线程。Web Worker可以创建多个实例，并且与主线程之间通过消息传递进行通信。

# 使用Web Worker

使用 Web Worker 非常简单，只需要遵循以下几个步骤：

  1. 创建一个新的 JS 文件，其中包含要在 Web Worker 中执行的代码。
  2. 在主线程中使用 new Worker() 构造函数创建一个 Web Worker实例，并指定要执行的脚本文件的 URL。
  3. 主线程通过 postMessage() 方法向 Web Worker 发送消息，Web Worker 通过onmessage事件监听器接收消息并进行相应的处理。
  4. Web Worker 执行完任务后，通过 postMessage() 方法将结果发送回主线程，主线程通过 onmessage 事件监听器接收并处理结果。

下面是一个简单的使用示例：

// 主线程main.js
const worker = new Worker('worker.js');
worker.postMessage('Hello from main thread!');
worker.onmessage = function (event) {
  console.log('Message received from worker thread:', event.data);
}

// worker.js
this.onmessage = function(event) {
  console.log('Message received from main thread:', event.data);
  // 执行耗时任务...
  // Web Worker 也通过相应的实现 API 与主线程进行通信
  this.postMessage('Hello from worker!');
};

# 应用场景

Web Workers 适用于许多应用场景，特别是需要处理大量数据或执行复杂计算任务的应用。一些常见的应用场景包括：

  • 图像处理：在 Web Workers 中执行图像处理操作，如图像滤镜、缩放、裁剪等。
  • 数据处理：使用 Web Workers 执行数据分析和处理任务，加快计算速度和提高数据处理效率。
  • 游戏开发：在游戏开发中使用 Web Workers 执行物理模拟、碰撞检测、人工智能计算等任务。
  • 大数据可视化：在大数据可视化应用中使用 Web Workers 处理大量数据并生成可视化图表。

# Web Worker 通信同步效率低与内存高

主线程与 Web Worker 通过 postMessage 通信的时候，data 会先被 copy 一份再传给 Web Worker；同样地，当 Web Worker 通过 postMessage 与主线程通信的时候，data 也会同样先被 copy 一份再传给主线程。 这样做显然会导致通信上同步效率低与占用内存过高问题，试想一下你需要在 Web Worker 里面解压一个 1G 大小的问题，你需要把整个 1G 的文件 copy 到 Web Worker 里，Web Worker 解压完这个 1G 文件后，再把解压完的文件 copy 回主线程里。这整个流程不仅仅耗时长，同步效率慢，而且还会导致CPU内存资源占用过高。

# SharedArrayBuffer

为了解决通信效率问题，浏览器提出了 ShareArrayBuffer，ShareArrayBuffer 基于 ArrayBuffer (opens new window) 和 TypedArray API (opens new window)。ArrayBuffer 对应一段内存（二进制内容），为了操作这段内存，浏览器需要提供一些视图（Int8Array 等），例如可以把这段内存当做每 8 位一个单元的 byte 数组，每 16 位一个单元的 16 位有符号数数组。

# 使用SharedArrayBuffer

下面是一个简单的使用示例：

// 主线程main.js
const worker = new Worker('worker.js');
// 创建一个4字节的共享内存区域
const sharedBuffer = new SharedArrayBuffer(1024);
// 创建一个Int8Array视图，用于操作共享内存
const intArrBuffer = new Int8Array(sharedBuffer);

for(let i=0;i<intArrBuffer.length;i++){
  intArrBuffer[i] = i;
}
console.log(intArrBuffer);

worker.postMessage(intArrBuffer);
worker.onmessage = function (event) {
  console.log('更改后的数据:', intArrBuffer);
}

// worker.js
this.onmessage = function(event) {
  const arrBuffer = event.data;
  // 执行耗时任务...
  arrBuffer[20] = 88;
  // Web Worker 也通过相应的实现 API 与主线程进行通信
  this.postMessage('Hello from worker!');
};

# 竞争问题（Race Conditions）

这段内存可以在不同的 Worker 之间共享，但是如果多个线程同时修改共享内存又会产生竞争问题（Race Conditions）： **计算机指令对内存操作进行运算的时候，我们可以看做分两步：一是从内存中取值，二是运算并给某段内存赋值。**当我们有两个线程对同一个内存地址进行 +1 操作的时候，假设线程是按先后顺序运行的，为了简化模型，我们可以如下图表示： 上面两个线程的运行结果也符合我们的预期，也即线程分别都对同一地址进行了 +1 操作，最后得到结果 3。但因为两个线程是同时运行的，往往会发生下图所表示的问题，也即读取与写入可能不在一个事务中发生： 这种情况就叫做竞争问题（Race Conditions）。

# Automics（原子操作）

原子操作

当内存被共享时，多个线程能够读写内存上的同一数据。原子操作会确保正在读或写的数据的值是符合预期的，即下一个原子操作一定会在上一个原子操作结束后才会开始，其操作不会被中断。

多线程只要存在，就会存在竞争，所以就必须有锁的概念。为了解决上述的竞争问题，浏览器提供了Atomics API (opens new window)，这组 API 是一组对 SharedArrayBuffer 和 ArrayBuffer 对象执行原子操作的静态方法，可以将读取和写入绑定起来。

# Automics API

  • Atomics.load(typedArray, index)：返回数组中指定位置的值。
  • Atomics.store(typedArray, index, value)：将给定的值存储在数组中的指定位置，并返回该值。
  • Atomics.wait(typedArray, index, value, timeout)：验证共享内存特定位置是否仍然包含给定值，如果是则休眠，直到被唤醒或超时。其返回一个内容为 "ok"、"not-equal" 或 "timed-out" 的字符串。
  • Atomics.notify(typedArray, index, count)：唤醒一些在等待队列中休眠的代理。
  • Atomics.add(typedArray, index, value)：将给定的值加到数组里的指定位置上，并返回该位置的旧值。此原子操作保证在修改后的值写回之前不会发生其他写操作。

# 解决竞争问题

例如下图中的 S1 到 S3 操作就被浏览器封装成 Atomics.add() 这个 API，从而解决竞争问题。

下面是一个简单的使用示例：

// 主线程main.js
const worker = new Worker('worker.js');
// 创建一个4字节的共享内存区域
const sharedBuffer = new SharedArrayBuffer(1024);
// 创建一个Int8Array视图，用于操作共享内存
const intArrBuffer = new Int8Array(sharedBuffer);

for(let i=0;i<intArrBuffer.length;i++){
  intArrBuffer[i] = i;
}
console.log(intArrBuffer);

worker.postMessage(intArrBuffer);
worker.onmessage = function (event) {
  console.log('更改后的数据:', intArrBuffer);
}

// worker.js
this.onmessage = function(event) {
  const arrBuffer = event.data;
  // 执行耗时任务...
  // Atomics.add() 静态方法会将给定的值加到数组里的指定位置上，并返回该位置的旧值。此原子操作保证在修改后的值写回之前不会发生其他写操作。
  Atomics.add(arrBuffer, 20, 1)
  // Web Worker 也通过相应的实现 API 与主线程进行通信
  this.postMessage('Hello from worker!');
};

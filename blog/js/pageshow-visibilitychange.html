# pageshow、pagehide、visibilitychange对比

# 一、前言

移动端开发，visibilitychange的兼容性的问题，给我们的开发带来了很大的困扰。举个🌰： 假设有A，B两个页面

A：商品列表页
B：商品详情页

某一天万恶的产品来了一个数据统计的需求。 产品：小王，这里有一个数据统计方面的需求，用户每次访问页面A或者当页面A从程序后台切换到前台的时候，需要进行一次PV的埋点上报。这个需求是大老板提出来的，好好干哦，今天必须弄好。 小王：一脸黑线，🤷‍♀️，嘴里mmp。搜了个搜，找到了浏览器的pageshow和visibilitychange事件，又查了下这两个事件的兼容性 (opens new window)，好家伙，96.68%，移动端几乎完全兼容，就是它了，开干。

# 二、定义

# pageshow/pagehide事件

pageshow/pagehide 指的是页面的进入与离开。

pageshow 事件在用户浏览网页时触发(onload之后)。类似于 onload 事件，但onload只在页面第一次加载时触发，pageshow 事件会在每次加载页面时触发，即 onload 事件当页面从浏览器缓存中读取时不触发(如通过前进/后退进入页面时)。

pagehide 事件在用户离开网页时触发(onunload之前)

# visibilitychange事件

visibilitychange 指的是页面的可见性。

visibilitychange会在页面被隐藏或显示时(如浏览器tab切换、窗口被最小化等)触发，配合document.hidden或者document.visibilityState的属性可以实现页面前台/后台切换的功能（比如切到后台时停止声音播放，切到前台时恢复播放"等功能）

# 三、测试对比及分析

# 对照测试

我们可以通过一段代码了解以上两者功能上的区别：

window.addEventListener('pageshow', () => {
    console.log('pageshow: 页面显示');
});
window.addEventListener('pagehide', () => {
    console.log('pagehide: 页面隐藏');
});

document.addEventListener('visibilitychange', () => {
    if (document.hidden) {
        console.log('visibilitychange: 页面隐藏');
    } else {
        console.log('visibilitychange: 页面显示');
    }
});

# 测试结论

# 1. 页面进入，包括刷新会触发pageshow，离开都会触发pagehide；

# 2. 页面前台/后台切换(选项卡切换)，只会触发 visibilitychange 显示与隐藏；

# 3. 前进和后退，有的浏览器（webkit内核）会保留之前的数据，有的则是直接刷新（比如chrome浏览器使用blink内核，前进后退直接刷新）。

# 4. 前进和后退，会保留之前访问数据的所有浏览器中：

TIP

  • 都会依次触发 pagehide，和 pageshow，但是有的浏览器会触发 visibilitychange，有的则不会；
  • ios手机上所有的浏览器都是使用webkit内核，前进后退都不会触发visibilitychange事件。
  • android手机上，有的前进后退会触发visibilitychange事件，比如华为自带浏览器，UC浏览器等；
  • android手机上，有的前进后退不会触发visibilitychange事件，比如百度APP``qq浏览器等；
  • android手机上，有的前进后退会打开一个新的webview，比如小米自带浏览器，这时候相当于页面切回前台/后台，也会触发visibilitychange事件。

大家若有兴趣，可以访问这里 (opens new window)感受下每个浏览器事件变化的触发。

# 原因分析

  1. 由于webkit内核中bf-cache的存在，于是便可以解释第3点，为什么有的浏览器会保留之前的数据了，在有bfcache的情况下，浏览器会保留之前的数据。

TIP

bf-cache，又称往返缓存，具体介绍可以查看我的另一篇文章bfcache问题及解决方案。

  1. 至于第4点，visibilitychange事件不同的表现，官方也给了详细解释，使用webkit内核的浏览器，会有这方面的bug。而且官方也不推荐，使用visibilitychange事件监听页面是否离开。

# 四、终极解决方案

了解了pageshow,pagehide以及visibilitychange等事件在浏览器中的表现之后，那么有没有一统江湖的解决方案呢？

必须有。业务上可以添加如下代码：

const _sendPageLoad = () => {
  if (document.visibilityState === 'visible' || !document.hidden) {
    <!-- TODO：上报PV埋点 -->
  }
}
const _listenPageShow = (event: Event) => {
  if (event.persisted) { // 从往返缓存加载
     <!-- TODO：上报PV埋点 -->
  }
}
const _listenPageHide = (event: Event) => {
  if (event.persisted) {
    document.removeEventListener('visibilitychange', _sendPageLoad)
  }
}
window.addEventListener('pageshow', _listenPageShow)
window.addEventListener('pagehide', _listenPageHide)
document.addEventListener('visibilitychange', _sendPageLoad)

onload(()=>{
  _sendPageLoad()
})
onbeforeunload(()=>{
  window.removeEventListener('pageshow', _listenPageShow)
  window.removeEventListener('pagehide', _listenPageHide)
  document.removeEventListener('visibilitychange', _sendPageLoad)
})
  • 监听onload，在页面第一次加载的时候触发埋点事件。
  • 监听pageshow事件，当页面返回触发缓存逻辑时，event.persisted为true，这时候重新上报埋点。
  • 监听visibilitychange事件，当页面显示时，document.visibilityState为visible，document.hidden为false，上报埋点
  • 监听pagehide事件，当event.persisted为true（当前页面被bfcache缓存时，不会触发beforeunload和unload），移除visibilitychange的事件监听，避免返回到当前页面时，由于不同浏览器的差异性，会再次触发visibilitychange事件，造成埋点的多报

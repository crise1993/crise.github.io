# 项目CI/CD持续集成部署方案，看这一篇就够了

# 概念

  • CI（Continuous Integration）

指的是持续集成，即项目代码的新更改会定期构建、测试并合并到代码仓库中，有效解决一次开发多个项目分支导致代码冲突问题。

  • CD（Continuous Delivery/Continuous Deployment）

指的是持续交付/持续部署，即项目代码的新更改可以自动或手动合并到主分支，并在合并至主分支后自动执行构建、测试流程，检测新更改是否对主分支代码产生影响。构建测试通过后，会自动发布并部署至生产环境，有效减轻运维团队负担。

# 导语

一个网站的开发上线往往都需要以下几个步骤：

TIP

编写代码 -> （单元测试/集成测试） -> 上传至代码仓库 -> 打包构建 -> 上传至服务器 -> 配置 Nginx/Apache 将 80 端口映射至网站文件夹

现代软件开发的需求加上部署到不同基础设施的复杂性使得创建应用程序成为一个繁琐的过程。当应用程序出现规模性增长，开发团队人员变得更分散时，快速且不断地生产和发布软件的流程将会变得更加困难。

为了解决这些问题，开发团队开始探索新的策略来使他们的构建、测试和发布流程自动化，以帮助其更快地部署新的生产。这就是持续集成和持续交付/部署发展的由来。

当我们有了 CI/CD 的系统之后，我们就只需要编写代码，剩下的步骤都交给 CI/CD 系统来处理，这极大地解放了我们的双手，提升了开发效率。

jenkins、Travis CI、Drone CI、Gitlab CI/CD等持续部署工具都可以很好的实现这一需求。本文通过实践（以部署到CenteOS服务器为样板），致力于比较不同方案的优劣。

# jenkins持续集成部署

TIP

Jenkins (opens new window) 是开源 CI&CD 软件领导者，提供超过 1000 个插件来支持构建、部署、自动化，满足任何项目的需要。

  • 优点：

    1. 编译服务和代码仓库分离，耦合度低
    2. 生态丰富，插件众多，支持语言众多，各种配置文档比较全
    3. 可视化流程，支持多语言项目构建，有统一的web管理界面
    4. 完全兼容ant、maven、gradle等多种第三方构建工具
    5. 同时跟svn、git能无缝集成，也支持直接与知名源代码托管网站，比如：GitHub、bitbucker直接集成

  • 缺点：

    1. 插件以及是自身安装较为复杂
    2. 依赖于Java，内存占用比较大，体量较大，不是很适合小型团队
    3. 使用webhook插件，里面配置的git仓库地址不能是127.0.0.1，必须得有仓库服务器（得基于gitlab搭建自身的仓库服务器）

jenkins具体配置教程可以参考这篇文章写给前端的 Jenkins 教程——10分钟实现前端/ Node.js 项目的 CI/CD (opens new window)。

通过Jenkins和gitlab仓库打通，配置gitlab仓库的hooks后，当我们的仓库代码push到指定的分支后,自动触发Jenkins任务，Jenkins会自动将最新代码pull到centos中,接下来我们需要在代码中执行npm install安装所以依赖,然后再对项目进行npm build打包，并更新到Nginx指定的目录中，重启Nginx服务，就实现了项目的持续部署。

# drone CI实现自动集成部署

TIP

drone (opens new window)是一个现代化的持续集成平台，使用强大的云原生管道引擎自动化构建、测试和发布工作流程， 它可以使用简单的YAML配置文件来定义和执行Docker容器中的Pipelines，与流行的源代码管理系统无缝集成

  • 优点：
    1. 配置较简单，只需要配置drone.yml文件
  • 缺点：
    1. 安装比较复杂，得配合docker进行安装
    2. Drone的文档更新不及时，文档有的地方是有问题的
    3. 可视化界面得基于Drone api进行二次封装，学习成本较高

drone ci具体配置教程可以参考这篇文章使用drone ci自动化部署静态博客 (opens new window)， drone的文档更新不及时，文档有的地方还是有问题的，具体配置坑比较多。

# Gitlab CI/CD实现（推荐）

Gitlab CI是gitlab8.0之后自带的一个持续集成系统，中心思想是当每一次push到gitlab的时候，都会触发一次脚本执行，然后脚本内容包括了测试、编译、部署等一系列自定义内容。

TIP

Gitlab CI的脚本执行，需要自定义安装对应Gitlab Runner来执行，代码push之后，webhook检测到代码变化，就会触发gitlab CI，分配到各个runner实例来运行相应的脚本script。这些脚本有的是测试项目用的，有的是部署用的。

GitLab Runner是配合GitLab CI/CD完成工作的核心程序，出于性能考虑，GitLab Runner应该与Gitlab部署在不同的服务器上（Gitlab在单独的仓库服务器上，GitLab Runner部署在web应用的服务器上）。GitLab Runner在与GitLab关联后，可以在服务器上完成诸如项目拉取、文件打包、资源复制等各种命令操作。

  • 优点：
    1. 轻量级，不需要复杂的安装手段
    2. 配置简单，使用YAML进行配置，与gitlab可直接适配，任何人都可以很方便的使用
    3. Gitlab与其CI平台紧密集成，可以方便查看每个打开或关闭拉取请求的运行和完成管道
    4. GitLab与GitLab CI的深度整合，权限可以统一管理
    5. GitLab CI 是Git存储库管理器GitLab的固定组件，因此CI/CD流程实时构建日志十分清晰，UI交互体验很好
    6. 天然支持分布式，gitlab的runner可以装在任何一台电脑上，方便测试和集成
  • 缺点：
    1. 没有统一的管理界面，无法统筹管理所有项目
    2. 配置依赖于代码仓库，耦合度没有Jenkins低
    3. 没有定时构建功能，尽管无法做到这一点，其实如果配置了提交代码既触发流水线，那么最后一次提交的构建在什么时候没有什么不同，反而减少未提交代码的定时构建资源浪费

Gitlab CI/CD具体配置教程可以参考这篇文章前端自动化部署：借助Gitlab CI/CD实现 (opens new window)。

安装gitlab-runner前，请检查web服务器是否安装了git,Node,以及项目中的npm包管理工具yarn或者pnpm。

1、runner 安装 (opens new window)

TIP

如果不清楚应该下载哪个，可以使用uname -a查看当前web服务器版本，下载完成之后，可以通过gitlab-runner --version查看当前gitlab-runner安装版本。

2、runner 注册 (opens new window)

TIP

记得需要关闭settings -> CI/CD -> Runners -> Shared runners中的Enable shared runners for this project选项，因为如果需要使用Shared runners实例的话，必须得自己有自己的gitlab服务器，或者实名注册gitlab仓库，并且绑定银行卡。

注册完成后，使用gitlab-runner verify查看当前runner实例是否激活，或者进gitlab项目仓库，settings -> CI/CD -> Runners -> Available specific runners，如果实例前面图片为绿色圈圈，代表激活成功，如果为三角形内感叹号，代表配置出了问题，这时候得检查下运行命令或者填写参数是否正确。如果配置不成功的话，后面运行pipeline会报各种各样的错误。

3、配置.gitlab-ci.yml (opens new window)

当然也可以结合docker (opens new window)更好的实现项目的打包部署。

# 结语

GitLab CI有助于DevOps人员，例如敏捷开发中，开发和运维是同一个人，最便捷的开发方式。

Jenkins CI适合在多角色团队，职责分明、配置和代码分离、插件丰富。

在使用过两者后，个人认为GitLab CI更简单易用，如果有Gitlab CI达不到的要求，可以考虑使用jenkins。
